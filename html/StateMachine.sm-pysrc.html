  2  """This module implements state machines for event-driven programming. 
  3   
  4  This module provides the class `StateMachine` and the class `Event`. 
  5  It also provides the classes `State` and `Transition`, although these are not normally used directly by applications. 
  6  A companion module, `decorator`, provides two decorators, ``@state`` and ``@transition`` that simplify the 
  7  definition of state machines. 
  8   
  9  State machines are a control structure particularly suited for *event-driven programming*, 
 10  i.e. when the program's behavior depends on events arriving asynchronously as is the case 
 11  with, e.g., user interfaces. 
 12   
 13  A state machine contains a set of *states*.  
 14  At any one time, only one state is *current*. If the current state is ``A``, the machine is said to be in state ``A``. 
 15  One state is the *initial state*, i.e. the current state when the machine is started. 
 16   
 17  Each state may have one or more *transitions* that describe how the machine reacts to events when it is in this state. 
 18  Each transition specifies the events it recognizes and a *destination state*. 
 19  When an event is processed by the state machine, it checks each transition of its current state. 
 20  The first transition that matches the event is *fired*, and its destination state becomes the current state. 
 21  If no transition matches the event, the event is ignored. 
 22   
 23  In this implementation of state machines, events that match a transition are specified by: 
 24   
 25          - an *event type*, which is usually a class object, e.g., ``ButtonPress`` for an event representing the user pressing the button of the mouse; 
 26          - optional extra arguments relative to the state of the event, e.g., ``'Button1'`` to specify which button of the mouse should be pressed; 
 27          - an optional *guard*, which is a predicate evaluated on the event. 
 28   
 29  States and transitions can have *actions* attached to them. 
 30  States can have an *enter action*, called when the state becomes current, 
 31  and a *leave action*, called when the state ceases to be current. 
 32  Transitions have a *transition action* that is called when the transition is fired. 
 33  In summary, if the current state is ``A`` and the transition being fired has destination state ``B``, 
 34  the following actions are called in that order: 
 35   
 36          - the leave action of the current state ``A``; 
 37          - the transition action; 
 38          - the enter action of the destination state ``B``, which is now the new current state. 
 39   
 40  State machines are traditionally represented by a graphical notation: 
 41  states are represented by circles and transitions by directed arrows going from their starting state to their destination state. 
 42  The initial state is usually marked with an L-shaped arrow or a thicker border. 
 43  Transitions are typically labelled by an expression of the form  
 44   
 45          ``Event(args) [& guard] [/ action]`` 
 46   
 47  that specifies the event type, additional arguments, optinal guard, and optional action.
